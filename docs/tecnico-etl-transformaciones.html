Generación de Llaves Surrogate con CRC32
¿Por qué CRC32 y no SEQUENCE o autoincrement?

Hive no tiene secuencias nativas ni autoincrement. El ETL sobreescribe las tablas en cada ejecución (mode="overwrite"), lo que hace inviable una secuencia externa.

Solución: CRC32 sobre el UUID del Proyecto 1 genera un entero de 32 bits determinístico — el mismo UUID siempre produce el mismo key. Esto permite que los jobs de hechos resuelvan las FK buscando CRC32(usuario_id) en la dimensión sin necesidad de un lookup adicional.

usuario_key = F.crc32(F.col("id").cast("string")).cast("bigint")

Limitación conocida: CRC32 produce colisiones en espacios muy grandes. Para los volúmenes del proyecto (< 100k registros por dimensión) la probabilidad de colisión es despreciable.
# Ejemplo: generación de usuario_key en cargar_dimensiones.py df_usuarios = spark.read.jdbc(JDBC_URL, "users", properties=JDBC_PROPS) dim_usuario = df_usuarios.select( F.crc32(F.col("id").cast("string")).cast("bigint").alias("usuario_key"), F.col("id").alias("usuario_id"), F.col("name").alias("nombre"), F.col("role").alias("rol"), F.to_date(F.col("created_at")).alias("fecha_registro") ) dim_usuario.write.mode("overwrite") \ .format("hive") \ .saveAsTable("restaurants_dw.dim_usuario")

Job 2 — cargar_fact_items_pedido.py
2
spark/jobs/cargar_fact_items_pedido.py
Construye la tabla de hechos principal uniendo order_items con orders, users y restaurants via CRC32.
PasoOperaciónDetalle
1. Lectura READ
Lee order_items y orders vía JDBC desde re2_postgres.
2. Join JOIN
Hace INNER JOIN de order_items con orders en order_id para obtener el timestamp del pedido y el user_id, restaurant_id.
items.join(orders, items.order_id == orders.id)
3. Surrogate keys CRC32
Calcula las 4 FK como CRC32 de sus respectivos UUIDs. El tiempo_key se deriva del created_at del order.
tiempo_key = year*1000000 + month*10000 + day*100 + hour
usuario_key = crc32(user_id)
restaurante_key = crc32(restaurant_id)
producto_key = crc32(product_id)
4. Medidas CALC
Calcula monto_total = cantidad × precio_unitario en Spark. Ambos campos vienen de order_items.
F.col("quantity").cast("int")
F.col("unit_price").cast("decimal(10,2)")
monto_total = cantidad * precio_unitario
5. Atributos deg. MAP
Mapea status del order (pending / confirmed / cancelled) y takeaway → es_para_llevar.
6. Escritura WRITE
Escribe en Hive con modo overwrite. Usa FileOutputCommitter v2 para commit directo sin staging.
df.write.mode("overwrite").format("hive").saveAsTable("restaurants_dw.fact_items_pedido")

Configuración Spark Compartida (via Airflow)
spark.driver.host = "ra-airflow" (con guión, no underscore).
Java URI (RFC 2396) rechaza underscores en hostnames, causando "Invalid Spark URL" al enviar el job. El hostname del contenedor Airflow está definido en docker-compose como hostname: ra-airflow.
# SPARK_CONF en restaurants_etl.py — compartida por todos los SparkSubmitOperator SPARK_CONF = { "spark.sql.catalogImplementation": "hive", "spark.sql.warehouse.dir": "/opt/hive/data/warehouse", "spark.hadoop.hive.metastore.uris": "thrift://hive-metastore:9083", # JDBC pre-bakeado en la imagen — evita descargas Maven en runtime "spark.jars": "/opt/airflow/jars/postgresql-42.7.3.jar", # Hostname del driver (el contenedor ra_airflow) "spark.driver.host": "ra-airflow", "spark.driver.bindAddress": "0.0.0.0", # FileOutputCommitter v2: commit directo sin staging del driver "spark.hadoop.mapreduce.fileoutputcommitter.algorithm.version": "2", }
